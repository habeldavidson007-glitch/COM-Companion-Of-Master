Merge recommendation
Do not merge yet — 3 blockers must be fixed first
The PR is purely additive (11 new files, zero changes to master's 22 files). Master stability is not at risk. But the PR introduces broken infrastructure that would create false confidence if merged unresolved.

Safe — no regressions possible

PR adds only src/com/ directory — 11 new files, zero edits to existing ones
Master's Tkinter UI, COMCore, tool_harness, all tools are untouched

Two model configs (qwen2.5 vs smollm2) with no switching layer — one is always wrong
auto_fix_missing_summaries() is a public stub — looks functional, does nothing
3 blockers before merge
1Fix the backlink key mismatch in WikiIndexer
add_backlink() stores raw path strings as keys. get_backlinks() transforms the path to a doc_id before lookup. They never match — the entire backlink graph is written correctly but permanently unreadable. Every orphan detection result, every concept graph traversal, every health report is wrong because of this single inconsistency.
indexer.py: standardize add_backlink() to use the same doc_id transform as get_backlinks()
2Wire WikiRetriever into com_chat.py — or don't ship it yet
The wiki layer is the entire purpose of this PR. After merging, a user runs com compile, populates 50 documents, and then asks COM a question. The LLM gets zero context from any of those documents. The wiki is a filing cabinet that no one ever opens. Either add the three lines of retrieval injection into process_query() before merging, or hold the PR until that is done. Shipping infrastructure without the wire-up is shipping half a feature.
com_chat.py: call WikiRetriever.search(query, top_k=3) and prepend snippets to system prompt
3Resolve the dual model config — pick one or add a config switch
Master targets qwen2.5:0.5b via direct Ollama calls in com_core.py. The PR's src/com/core/client.py hardcodes smollm2:1.7b-instruct-q4_K_M. After merging, both layers are active with different models, different token limits, and no user-facing way to choose. The README says smollm2 is the new default, but the running system still calls qwen2.5. A single MODEL constant in a shared config file, or an environment variable, resolves this completely.
config.py: MODEL = os.getenv("COM_MODEL", "smollm2:1.7b-instruct-q4_K_M")
What to do
RecommendedFix the 3 blockers in the PR branch, then merge
The fixes are small — the backlink bug is a one-line change, the wiki wire-up is about 8 lines, the model config is 3 lines. None require redesign. The PR's architecture is sound: SafeIO, SignalParser, and WikiRetriever are production-quality modules that genuinely improve COM. They just need to be connected. Estimated work: 1–2 hours. After that, this PR is a clear improvement over master and safe to merge.
Not recommendedMerge now as-is
Technically safe — master won't break. But the wiki system will be present, documented, and nonfunctional. Users who read the README will try com compile and com search, see them work, and then notice that COM still gives context-blind answers. That is worse than the feature not existing yet, because it creates the expectation of RAG without delivering it.
Not recommendedClose the PR and keep master as-is
Master has its own existing bug: the @GDT token is in VALID_PREFIXES and the harness GODOT system prompt instructs the LLM to output @GDT signals — but harness.py dispatches @GDT to a stub while tool_harness.py only has 'GODOT' as the executor key. Closing the PR and staying on master doesn't eliminate technical debt, it just freezes it.
