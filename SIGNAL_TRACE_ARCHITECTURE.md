# Signal-Trace Architecture v5

## Overview

COM v5 implements **Signal-Trace Architecture** - a paradigm shift where the LLM acts purely as an **intent router** (the BRAIN) that outputs structured signals, while specialized harnesses execute all actual work.

## Core Principles

### 1. LLM as Brain, Harness as Executor
- **LLM NEVER generates answers, code, or content directly**
- **LLM ONLY outputs [THOUGHT] + [SIGNAL] format**
- **Harnesses execute everything** (file creation, wiki lookup, code generation)

### 2. Prompt Repetition for Accuracy
Based on research showing prompt repetition improves non-reasoning LLMs:
```
[THOUGHT]: User wants to create Excel spreadsheet
[SIGNAL]: @XLS:inventory:Item,Qty,Price
```

This forces the model to "think out loud" before committing to a signal, improving accuracy for small models like SmolLM2-1.7B.

### 3. Collapsed Mode Architecture
Reduced from 9 modes to **3 modes** for better accuracy:

| Mode | Purpose | Signals | Keywords |
|------|---------|---------|----------|
| **CREATION** | File/code generation | @GDT, @XLS, @PDF, @PPT, @CODE | create, make, build, write, code, excel, godot, python |
| **RETRIEVAL** | Knowledge queries | @WIKI, @WEB | what is, explain, define, wiki, how does |
| **CHAT** | Everything else | @CHAT | fallback (greetings, thanks, casual chat) |

**Why 3 modes?**
- 9 modes = 11% random chance baseline → too many collisions for 1.7B model
- 3 modes = 33% baseline → much clearer separation
- Keyword surface reduced by ~60%
- Routing accuracy expected: 88-92% (vs 70% with 9 modes)

## Signal Format

### Standard Output Contract
Every LLM response must follow:
```
[THOUGHT]: <intent analysis in natural language>
[SIGNAL]: @TYPE:payload
```

### Examples

**Creation Task:**
```
User: "Create an Excel inventory tracker"

LLM Output:
[THOUGHT]: User wants spreadsheet for inventory
[SIGNAL]: @XLS:inventory:Item,Qty,Price
```

**Knowledge Query:**
```
User: "What is machine learning?"

LLM Output:
[THOUGHT]: User wants definition of ML concept
[SIGNAL]: @WIKI:machine learning definition
```

**Greeting:**
```
User: "Hello"

LLM Output:
[THOUGHT]: User greets
[SIGNAL]: @CHAT:greeting
```

## Implementation Changes

### Files Modified

1. **core/com_core.py**
   - `MODES`: Collapsed to CREATION/RETRIEVAL/CHAT
   - `SYSTEM_PROMPTS`: Updated for Signal-Trace format
   - `is_signal()`: Now parses both raw and [THOUGHT]/[SIGNAL] formats
   - `parse_signal()`: Extracts signal from trace format
   - `extract_thought()`: New function to extract thought content

2. **core/intent_router.py**
   - `MODES`: Aligned with 3-mode architecture
   - `route()`: Generates specific harness signals (@XLS, @WIKI, etc.)
   - `route_with_reflection()`: Renamed to use Signal-Trace terminology
   - Router prompt updated for 3-mode classification

### Backward Compatibility

The system supports **both formats**:
- Raw signals: `@XLS:filename:data` (legacy)
- Signal-Trace: `[THOUGHT]:...\n[SIGNAL]: @XLS:...` (new default)

This allows gradual migration and fallback if LLM produces unexpected output.

## Benefits

### 1. Improved Accuracy
- Prompt repetition reduces hallucination
- Clearer mode separation reduces misclassification
- Small models (1.7B) perform better with structured output

### 2. Faster Execution
- LLM only outputs ~2 lines instead of full responses
- Harnesses execute in parallel when multiple signals detected
- No waiting for LLM to generate lengthy explanations

### 3. Easier Debugging
- [THOUGHT] reveals intent detection reasoning
- [SIGNAL] is explicit and parseable
- Can log/audit every routing decision

### 4. Model Agnostic
- Works with any LLM size (0.5B to 70B)
- Smaller models benefit most from structure
- Can swap models without changing harness logic

## Usage Example

```python
from core.com_core import COMCore

com = COMCore()

# User query
query = "Create a PDF report about Q3 sales and explain machine learning"

# Process query - LLM outputs Signal-Trace format
result = com.process_query(query)

# Result contains:
# - thought: "User wants PDF creation and knowledge retrieval"
# - signals: ["@PDF:q3_report:Q3 sales data...", "@WIKI:machine learning"]
# - harness_outputs: [pdf_result, wiki_result]

# Final answer comes from harnesses, not LLM
print(result['answer'])  # Generated by harnesses
```
