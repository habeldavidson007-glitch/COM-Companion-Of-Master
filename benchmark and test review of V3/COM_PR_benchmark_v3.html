Confirmed bugs
Critical@GDT signal always fails — health key mismatch
In tool_harness.py, the health checker registers Godot as "GODOT" and the executors map uses "GODOT" as the key. But the signal token from the LLM is @GDT, which extracts to tool_type="GDT". is_tool_available("GDT") returns False because there is no "GDT" entry. The executor lookup also fails for the same reason. Fix: add 'GDT': execute_godot to the executors map and 'GDT': health_checker.tool_status['GODOT'] as an alias in the health checker — or rename the canonical key to "GDT" throughout.
CriticalWikiIndexer.get_backlinks() never returns results
add_backlink(source, target) stores the target as a raw path string key in self.index["backlinks"]. But get_backlinks(path) transforms the path to a doc_id via path.replace("/","_").replace(".md","") before lookup. The key stored is "wiki/godot_basics.md"; the key looked up is "wiki_godot_basics". They never match. The entire backlink graph is written correctly but permanently unreadable. Fix: either standardize on raw paths as keys throughout, or apply the same doc_id transform in add_backlink().
HighSignalParser accepts @@XLS (double-@, leading garbage)
The regex @(\w+):([^\s:]+)(?::([^\s]*))? uses finditer without anchoring. It finds @XLS as a substring inside @@XLS, treating the double-@ as valid. Any accidental duplication or string prefix contamination results in a parsed-but-wrong signal. Fix: require that the @ is preceded by a word boundary or start-of-string: (?<![\w@])@(\w+):.
HighWikiRetriever silently truncates documents to 2000 chars
self.documents[path] = content[:2000] is a hardcoded load-time truncation. For a 5000-word technical document, the last 60–70% never enters the TF-IDF index. A Godot physics document where the key term "CharacterBody2D" appears in the second section would have zero weight for that term. The fix is to index the full content for TF-IDF computation, and store only the first 200–400 chars as a preview snippet separately.
HighTwo model configs with no switching layer
Legacy core/com_core.py builds its Ollama call with qwen2.5:0.5b parameters. The new src/com/core/client.py hardcodes smollm2:1.7b. There is no shared config, no environment variable, no CLI flag to choose between them. If both are active simultaneously, they silently target different models with different token limits. The README says the default is smollm2, but the runtime core is still wired to qwen2.5. One of them is always wrong.
Mediumauto_fix_missing_summaries() is a public stub
HealthChecker advertises this method in its interface and the README implies health auto-repair works. The body iterates missing summaries, prints the path, and returns 0. No regeneration happens. A user running com health and seeing "3 documents with missing summaries" who then calls auto_fix gets silent zero-result behaviour with no error. Fix: either implement it or raise NotImplementedError and remove it from the public surface.
Mediumcompile_all(incremental=False) is identical to incremental=True
The for raw_path in raw_files loop calls self.compile_file(raw_path) if incremental else self.compile_file(raw_path) — both branches are the same call. Force-recompile does nothing different. This means the com compile --force CLI flag documented in the README is functionally broken.
LowWikiRetriever and SignalParser not wired to any runtime path
com_chat.py imports and calls COMCore but never calls WikiRetriever.search() before dispatching to the LLM. The retrieved wiki context that would make smollm2 informed is never injected. SignalParser is not called by tool_harness — PayloadValidator duplicates a subset of its logic without using the richer Signal dataclass. Both modules are fully functional but orphaned from the user-facing pipeline.

Demanding critique
Unacceptable regression across three versions
The @GDT mismatch has been present since v1 and still isn't fixed
The signal token is @GDT. The health checker key is "GODOT". The executor map key is "GODOT". These three strings have never agreed. Every single Godot file generation call — which is one of the two core use cases of the entire system — silently returns "Tool GDT is not available". This isn't an edge case. It's the primary user action. It has survived v1 review, v2 review, and now a PR review. A one-line alias fix is all it takes. The fact that it persists suggests the harness has never been run end-to-end with a real Godot prompt.
Architecture shipped before integration
The wiki system is complete infrastructure going nowhere
Five new modules were written, tested, and packaged — WikiCompiler, WikiIndexer, WikiRetriever, WikiHealthChecker, and the CLI. They work. But com_chat.py never calls WikiRetriever.search(). The LLM receives zero context from the knowledge base. The SignalParser is not used by tool_harness. The entire PR adds infrastructure but does not move the needle on the user's actual experience — the LLM is still as context-blind as it was in v1. Shipping infrastructure before the wiring that makes it useful is backwards. The integration — three function calls and a prompt injection — should have been the first commit, not deferred.
Technical debt compounding
Two model configs, two harness implementations, two validation systems
Legacy core/ targets qwen2.5. src/com/core/client.py targets smollm2. There is no config file, no environment variable, no single source of truth for which model is active. harness.py and tools/tool_harness.py both exist and solve overlapping problems. PayloadValidator and SignalParser both validate signals with different logic. Every duplication is a future divergence waiting to happen. A system that is already splitting at three seams — model config, harness dispatch, signal validation — needs consolidation before the next feature, not after.
The stub is worse than the absence
auto_fix_missing_summaries() should not exist in its current state
A method that is documented, named, and callable — but does nothing — is actively harmful. It creates false confidence. A developer runs it, gets no output, assumes it worked, and moves on. The missing summaries persist. The README says "Re-run compiler to regenerate summaries" as a suggestion — but the method that supposedly does this is a no-op. Either implement it in 15 lines (iterate missing paths, find raw source, call _generate_summary(), write back) or delete it from the public surface and raise NotImplementedError so callers fail loudly.
Genuine strength worth protecting
The SafeIO and TF-IDF retriever are production-quality
SafeIO's path traversal prevention via Path.resolve() comparison is the right approach — no regex, no blocklist, just filesystem semantics. WikiRetriever's TF-IDF with normalized term frequency is mathematically correct and blazingly fast (100-doc index in under 1ms). These two modules are the best code in the entire codebase. The 2000-char truncation fix is the only thing standing between them and being fully production-ready. Don't let future feature pressure compromise these foundations.
Systemic suggestion
The three-line integration that unlocks the entire wiki layer
In com_chat.py or COMCore.process_query(), before calling the LLM: instantiate WikiRetriever, call search(user_query, top_k=3), format the results as a context block, and prepend it to the system prompt. That is the entire integration. Every module you built this PR supports that pipeline. Without those three lines, the PR is infrastructure without a roof. The ambiguity resolver, TF-IDF ranking, concept graph, health monitoring — all of it becomes genuinely useful the moment context reaches the LLM.
